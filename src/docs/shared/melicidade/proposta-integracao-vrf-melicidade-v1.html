Integração Nativa do Sistema VRF

Entendimento da demanda, escopo técnico e proposta comercial para a substituição da integração atual via iframe/túnel por uma integração nativa do sistema de ar condicionado VRF Daikin, com resiliência offline, relatórios e agendamentos gerenciados diretamente pelo MyIO.

📖 Glossário

VRF Variable Refrigerant Flow — sistema de ar condicionado central onde uma unidade externa controla múltiplas unidades internas de forma independente por ambiente.
DCPF04 Controlador central Hitachi Reiri for Office Touch — o "cérebro" do sistema VRF instalado na Melicidade. Gerencia todas as unidades internas Daikin.
DCPA01 Adaptador de protocolo Daikin → Modbus RTU. Traduz o barramento proprietário DIII-Net das UIs Daikin para Modbus, permitindo que o DCPF04 as controle.
Modbus RTU/TCP Protocolo industrial aberto e amplamente documentado. Interface nativa do DCPF04 para integração com sistemas externos — independe da UI web.
Gateway MyIO Agente de software (Node-RED) instalado on-premise na Melicidade. Faz a ponte entre o DCPF04 (Modbus) e o ThingsBoard (nuvem MyIO).
Buffer local Armazenamento local no Gateway. Garante que dados coletados não se percam em quedas de conectividade — sincronizados quando a conexão volta.
ThingsBoard Plataforma IoT onde vivem os dashboards, relatórios e regras de automação do MyIO.
iframe / túnel Modelo atual: a central MyIO age como proxy reverso, exibindo a UI web do sistema externo dentro do dashboard. Dependente de conectividade contínua.

⚡ Por que é possível agora

O DCPF04 expõe Modbus RTU via RS485 — protocolo industrial aberto, amplamente documentado. O manual técnico completo está disponível (INS_DCPF04 Reiri for Office Touch). As unidades Daikin são conectadas via adaptador DCPA01 (DIII-Net → Modbus), que já faz a tradução internamente. Para o Gateway MyIO, basta ler/escrever registros Modbus — independente da UI web do Hitachi.

Se o DCPF04 expor Modbus TCP (porta 502 via Ethernet), não há necessidade de cabeamento adicional — a integração ocorre inteiramente via rede local existente.

✅ Resiliência Offline

O Gateway armazena dados localmente (buffer SQLite) durante quedas de conectividade. Quando a conexão é restabelecida, o backlog é sincronizado automaticamente para o ThingsBoard com os timestamps originais — sem lacunas no histórico e sem perda de dados.

⚠️ Principal Incógnita — Fase 0

O prazo total depende de dois fatores confirmados na Fase 0: (1) se o DCPF04 expõe Modbus TCP (porta 502) — se sim, sem necessidade de cabeamento adicional; se não, instalar adaptador USB-RS485 (+3–5 dias). (2) disponibilidade do mapa de registros Modbus do DCPA01 — se não disponível, descoberta ativa por varredura (+1 semana).

💡 Modelo de Contratação Sugerido

Sugerimos contratação por fase, com entrega e aceite ao final de cada etapa. A Fase 0 (levantamento) é o pré-requisito para refinar o escopo e confirmar o esforço das fases seguintes. Isso reduz risco para ambos os lados e garante visibilidade total do progresso.

⚠️ Premissas desta proposta

Estimativas baseadas em: (1) DCPF04 acessível via rede local; (2) Modbus TCP disponível na porta 502 — caso contrário, adaptador RS485 adicional com custo de hardware (~R$ 150) e +3 dias de instalação; (3) mapa de registros Modbus disponível ou fornecido pela Hitachi. Variações confirmadas na Fase 0 serão comunicadas antes do início das fases seguintes.
