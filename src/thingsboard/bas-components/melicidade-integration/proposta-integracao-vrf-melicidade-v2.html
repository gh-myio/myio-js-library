⚡ Por que é possível agora

O DCPF04 expõe Modbus RTU via RS485 — protocolo industrial aberto, amplamente documentado. O manual técnico completo está disponível (INS_DCPF04 Reiri for Office Touch). As unidades Daikin são conectadas via adaptador DCPA01 (DIII-Net → Modbus), que já faz a tradução internamente. Para o Gateway MyIO, basta ler/escrever registros Modbus — independente da UI web do Hitachi.

Se o DCPF04 expor Modbus TCP (porta 502 via Ethernet), não há necessidade de cabeamento adicional — a integração ocorre inteiramente via rede local existente.

💡 Princípios da proposta

(1) Offline-first: operação continua sem internet. (2) Zero invasão: não altera arquitetura Daikin/Hitachi, preservando garantias. (3) Dados no MyIO: histórico próprio, relatórios nativos. (4) Controle contínuo: SSO ThingsBoard, sem login duplo.

Seção 5 Dev Gateway

Desenvolvimento do software embarcado do Gateway MyIO

Plano detalhado do stack Node-RED + SQLite + cron + drivers Modbus que roda on-premise no Gateway. Este software é o coração da integração: coleta, armazena, agenda e sincroniza os dados do VRF com o ThingsBoard — em paralelo ao desenvolvimento dos painéis (Pág. 5).

🧩 Componentes do Software Embarcado

  • Node-RED Flows — orquestrador principal: polling, write RPC, scheduler, error handling, observabilidade
  • Driver Modbus TCP — cliente para DCPF04 via node-red-contrib-modbus (porta 502)
  • Driver Modbus RTU — fallback via USB-RS485 (contingência se Modbus TCP indisponível)
  • Buffer SQLite — fila de telemetria com retention policy e rotação automática (não enche disco)
  • Cron Scheduler Local — execução de agendamentos offline-first (independe de internet)
  • MQTT Client — sync para ThingsBoard via MQTTS (porta 8883 com TLS + certificado)
  • RPC Handler — recebe comandos do TB (on/off, setpoint, modo) e traduz para Modbus write
  • Health Check / Heartbeat — ping periódico + alarme MyIO se Gateway sumir
  • Watchdog Hardware — reinício automático em travamento de serviço
  • VPN WireGuard Client — canal seguro de suporte remoto (sem SSH público)

🔌 Interfaces & Protocolos

  • Downstream (LAN): Modbus TCP (porta 502) ↔ DCPF04 — leitura FC03/FC04 + escrita FC06/FC16
  • Upstream (WAN): MQTTS (porta 8883) ↔ ThingsBoard — telemetria, RPC e heartbeat
  • Config management: provisionamento via arquivo JSON + hot-reload de mudanças (sem restart)
  • Audit log: registro local de todo comando write Modbus — rastreabilidade total
  • API local de status: endpoint HTTP para health check na LAN (opcional)
  • Vault criptografado: persistência segura de credenciais (não hardcoded em flows)
  • Failover 4G/5G: detecção automática + switch transparente para link móvel

🧪 Qualidade, Testes & Homologação

  • Unit tests: funções puras (parsers, transformadores de registros Modbus) — Vitest/Jest
  • Integração: simulador Modbus local (pymodbus) + TB de desenvolvimento
  • Benchmark: polling sustentado (50+ devices a cada 60s) sem perda de pacotes
  • Failover: desconexão forçada (LAN + WAN) + validação de sync ao reconectar
  • Buffer overflow: simular 24h offline + confirmar que rotação SQLite funciona
  • Segurança: teste de hardening OS + revisão de portas expostas (pentest interno)
  • Homologação: bateria de testes assinada pelo cliente antes do Go-Live

🚀 Sprint 0 — Kickoff & Fundação Técnica (Gateway)

  • Kickoff: alinhamento com time de frontend (painéis VRF), contratos MQTT/RPC compartilhados
  • System Architect: topologia de flows Node-RED, padrões de erro, política de retry
  • API Design: contratos MQTT (telemetria, RPC, heartbeat), versionamento, payloads JSON
  • Modelagem de Dados: schema do buffer SQLite (retention, índices), estrutura de registros Modbus
  • Documentação: arquitetura do Gateway, runbook operacional, guia de troubleshooting
  • Prototipagem: POC — ler 1 registro Modbus do DCPF04 simulado e enviar ao TB de dev
  • Critérios de Aceite: DoD por sprint, métricas (coverage ≥80%, benchmark sustentado, uptime >99%)

Seção 6 Gateway

O que é o Gateway MyIO on-premise

Pequeno computador (edge gateway industrial) instalado dentro da Melicidade, na mesma LAN do DCPF04. Roda o stack de software MyIO (Node-RED + buffer SQLite) e faz a ponte entre o controlador VRF e a nuvem ThingsBoard. Não realiza acionamento elétrico direto sobre o VRF: toda a interação ocorre por rede (Modbus TCP/Ethernet) com o DCPF04, que mantém integralmente a responsabilidade física pelo barramento Daikin (DIII-Net via DCPA01).

🖥️ Composição de Hardware

  • SBC industrial ARM quad-core fanless em case DIN-rail/rack — Raspberry Pi 4/5, Orange Pi 5 ou Radxa ROCK 5 (referência: plataforma padrão MyIO para gateways)
  • GPIO header 40 pinos — requisito do rádio MyIO (SPI/UART/I²C) para integração com o ecossistema de sensores do ecossistema MyIO
  • 4–8 GB RAM LPDDR4, armazenamento 64+ GB (eMMC ou SSD NVMe via PCIe/M.2)
  • Ethernet Gigabit nativa + 2ª porta via adapter USB 3.0 — permite segmentação (1× LAN Modbus DCPF04 · 1× WAN) por segurança
  • Modem 4G/5G USB (com slot SIM) — failover de internet automático
  • WiFi + Bluetooth nativos (provisionamento local · não usados em produção)
  • Alimentação: 5V DC (USB-C PoE HAT opcional) · TDP ≤15W · Fanless (dissipador passivo)
  • Footprint: ~1U de rack ou DIN-rail em quadro técnico · Operação 24/7

📶 Topologia de Conectividade

O Gateway tem duas direções de comunicação independentes:

🛡️ Garantias do Fabricante Preservadas

A integração MyIO não invade nem altera a arquitetura elétrica/eletrônica original Daikin/Hitachi. Nenhum cabo é cortado, nenhum sensor é interceptado, nenhum sinal físico é injetado no barramento DIII-Net. O Gateway é apenas um cliente Modbus TCP do DCPF04 — uso documentado e suportado pela própria Hitachi no manual do equipamento (INS_DCPF04 Reiri for Office Touch).

⚡ Sem I/O Físico no VRF

O Gateway MyIO não tem nenhum I/O físico (digital, analógico, relé, 0–10V, 4–20mA) conectado ao VRF. Possui apenas conexão de rede Ethernet (RJ45).

[UIs Daikin VRF] │ DIII-Net (proprietário Daikin) ▼ [DCPA01] ← adaptador físico DIII-Net→Modbus │ RS485 (par trançado, A/B/GND) ▼ [DCPF04 Reiri] │ Ethernet RJ45 — Modbus TCP/502 ▼ [Gateway MyIO] ← só RJ45, nada físico no VRF │ Ethernet RJ45 — MQTT/HTTPS ▼ [ThingsBoard Cloud]

🔁 Entrada e Saída — apenas via Modbus

Apesar de não ter eletrônica de campo, o Gateway faz leitura e escrita — tudo encapsulado em Modbus TCP sobre a rede:

DireçãoOperação ModbusExemplos
Entrada (read) FC03 / FC04 Temperatura ambiente, setpoint, status on/off, código de erro, modo
Saída (write) FC06 / FC16 Ligar/desligar zona, mudar setpoint, trocar modo, fan speed

A "saída" do Gateway não é um pino que aciona algo — é um comando de rede que escreve num registro do DCPF04. Quem fisicamente comanda a UI Daikin é o próprio DCPF04 via DCPA01.

Seção 7 Infra

Três pilares de infraestrutura

Especificação completa para garantir operação 24/7 da integração VRF — com camadas de proteção e redundância proporcionais à criticidade do sistema de climatização para a operação do empreendimento.

🔄 Redundância

  • Energia: no-break (UPS) ≥ 30 min autonomia
  • Internet: RJ45 primário + 4G/5G failover automático
  • Buffer local: SQLite persiste todas as leituras em quedas
  • Cloud: ThingsBoard MyIO em cluster HA
  • Hardware spare: 1 Gateway extra cold standby (recomendado)
  • Time sync: múltiplos NTPs (pool + fallback local)

🔒 Segurança

  • Segmentação: Modbus em VLAN isolada da rede corporativa
  • Firewall outbound-only: zero portas expostas à internet
  • TLS/MQTTS (porta 8883 com certificado) para sync ThingsBoard
  • VPN WireGuard MyIO para suporte (sem SSH público)
  • OS hardenizado: Debian minimal + fail2ban + auto-update
  • APN privada se 4G/5G — sem IP público
  • Audit trail: log de todo comando write Modbus
  • Credenciais em vault (não hardcoded em flows)

Glossário — termos técnicos usados nesta proposta

Referência rápida dos termos, protocolos, equipamentos e conceitos de infraestrutura mencionados ao longo do documento. Organizado em dois grupos: (1) sistema VRF e comunicação com o DCPF04; (2) Gateway, rede, segurança e plataforma.

🏭 Sistema VRF & Comunicação

VRF Variable Refrigerant Flow — sistema de ar condicionado central onde uma unidade externa controla múltiplas unidades internas de forma independente por ambiente.
DCPF04 Controlador central Hitachi Reiri for Office Touch — "cérebro" do sistema VRF instalado na Melicidade. Gerencia todas as unidades internas Daikin.
DCPA01 Adaptador de protocolo Daikin → Modbus RTU. Traduz o barramento proprietário DIII-Net das UIs Daikin para Modbus.
DIII-Net Barramento proprietário Daikin para comunicação entre unidades externas e internas do VRF — não é publicamente documentado.
Modbus RTU/TCP Protocolo industrial aberto e amplamente documentado. Interface nativa do DCPF04 — RTU usa RS485; TCP usa Ethernet (porta 502).
FC03/FC04/FC06/FC16 Function Codes Modbus: FC03/FC04 leem registros (entrada); FC06 escreve 1 registro, FC16 escreve múltiplos (saída).
RS485 Padrão elétrico industrial para transmissão serial diferencial em par trançado (A/B/GND). Meio físico usado pelo Modbus RTU.
SMeter Medidor elétrico Daikin opcional por zona. Se instalado, permite coletar potência (W) e consumo (kWh) por ambiente.
iframe / túnel Modelo atual: a central MyIO age como proxy reverso, exibindo a UI web Hitachi dentro do dashboard. Dependente de conectividade contínua.

🌐 Gateway, Rede, Segurança & Plataforma

Gateway MyIO SBC industrial ARM (Raspberry Pi 4/5, Orange Pi 5 ou similar) fanless on-premise que faz a ponte entre o DCPF04 (Modbus TCP) e o ThingsBoard (nuvem MyIO). Possui GPIO 40 pinos para o rádio MyIO (integração com ecossistema de sensores). Sem I/O elétrico no VRF.
Node-RED Runtime de fluxos visuais em Node.js — orquestra o polling Modbus, o buffer local e o sync para a nuvem no Gateway MyIO.
Buffer local (SQLite) Banco embarcado no Gateway que armazena leituras durante quedas de conectividade — sincronizado quando a internet volta, sem perder timestamps.
MQTT / MQTTS Protocolo leve de mensageria IoT. MQTTS = MQTT sobre TLS (porta 8883). Usado para sync Gateway ↔ ThingsBoard.
ThingsBoard Plataforma IoT onde vivem os dashboards, relatórios, alarmes e Rule Engine (agendamentos) do MyIO.
Rule Engine Motor de regras declarativas do ThingsBoard — dispara ações (alarmes, RPC, notificações) em resposta a eventos de telemetria.
VLAN Segmentação lógica de rede num switch gerenciado — isola o segmento Modbus da rede corporativa por segurança.
WireGuard VPN Protocolo moderno de VPN (rápido e seguro) usado pela MyIO para suporte remoto ao Gateway — sem expor SSH à internet.
UPS / No-break Uninterruptible Power Supply — mantém o Gateway operando em quedas curtas de energia e protege contra picos/sub-tensão.
M2M IoT (4G/5G) Plano de dados móvel específico para telemetria 24/7 de baixo volume — preço melhor que linha móvel convencional, suporta APN privada.
Failover / Cold Standby Failover: troca automática para via secundária (ex: 4G assume se RJ45 cair). Cold standby: hardware spare desligado em prateleira, restaurado sob demanda.
SLA / CAPEX / OPEX SLA: Service Level Agreement (tempo de resposta contratual). CAPEX: investimento único (hardware). OPEX: custo recorrente (licenças, planos).
FTE / FTE-mês Full-Time Equivalent — unidade de esforço equivalente a 1 profissional dedicado integralmente. "FTE-mês" = 1 pessoa × 1 mês de trabalho em tempo integral (~160h). Usado para dimensionar alocação e custo de projeto independentemente de quantas pessoas o executam.

Análise hipotética — incorporação futura do CHILLER após V1 estabilizado

Simulação de esforço, prazo, atividades e alocações caso o cliente decida incorporar o sistema CHILLER (Niagara Framework via BACnet/IP) em uma V2 do projeto. Estimativa baseada em máximo reaproveitamento da base V1 VRF já entregue — infra, imagem do Gateway, plataforma, widgets e padrões de código.

♻️ O que é Reaproveitado do V1 (zero retrabalho)

  • Infraestrutura física (100%) — Gateway hardware, spare, UPS, switch L2, cabos, modem 4G/5G, rack, cabeamento já instalados. CAPEX adicional: R$ 0.
  • Imagem base do Gateway (100%) — Debian minimal + Node-RED + SQLite + MQTT + cron + VPN WireGuard + watchdog já provisionados e em operação.
  • Plataforma ThingsBoard (100%) — instance TB, AlarmServiceOrchestrator, Rule Engine patterns, MyIOOrchestrator, MenuView, padrões shadow DOM.
  • Framework de widgets (70–90%) — Card Zona, Widget Status, Widget Alarmes, Widget Agendamentos, AllReportModal — só requerem adaptação de datasource e labels.
  • Framework de testes (100%) — Vitest setup, simuladores, CI/CD pipelines, checklist de homologação.
  • Documentação base (60%) — runbook, arquitetura geral, padrões de deploy — adaptar só as partes BACnet-específicas.
  • Equipe treinada (100%) — time já conhece o cliente, a infra e os padrões — onboarding zerado.

🆕 Escopo NOVO (CHILLER-específico)

  • Driver BACnet/IP para comunicação com Niagara Framework (JACE) — node-red-contrib-bacnet ou implementação custom
  • Discovery Niagara — varredura UDP 47808, descoberta de points, mapeamento via Workbench Tridium
  • Widget Temperatura CHILLER — inlet water, outlet water, Δ water (diferente do simples temp ambiente do VRF)
  • Widget Vazão (Flow) — fluxo de água gelada em L/min com curva histórica
  • Widget Pressão — pressão de sucção e descarga do compressor
  • Widget Compressor — status, % de carga, runtime, ciclos liga/desliga
  • Widget Energia CHILLER — kWh específico do chiller (via 3F MyIO se disponível)
  • Catálogo de alarmes Niagara — mapeamento dos códigos BACnet/Niagara → Rule Engine TB
  • Automação específica — lógicas de sequenciamento de compressores, chilled water setpoint reset
  • Testes E2E com Niagara station real — validação com a estação Mercado_Livre do cliente
