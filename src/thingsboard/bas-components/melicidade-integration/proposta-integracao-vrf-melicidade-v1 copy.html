⚡ Por que é possível agora

O DCPF04 expõe Modbus RTU via RS485 — protocolo industrial aberto, amplamente documentado. O manual técnico completo está disponível (INS_DCPF04 Reiri for Office Touch). As unidades Daikin são conectadas via adaptador DCPA01 (DIII-Net → Modbus), que já faz a tradução internamente. Para o Gateway MyIO, basta ler/escrever registros Modbus — independente da UI web do Hitachi.

Se o DCPF04 expor Modbus TCP (porta 502 via Ethernet), não há necessidade de cabeamento adicional — a integração ocorre inteiramente via rede local existente.

Gateway MyIO — Especificação Técnica
Seção 2 Gateway

O que é o Gateway MyIO on-premise

Pequeno computador (edge gateway industrial) instalado dentro da Melicidade, na mesma LAN do DCPF04. Roda o stack de software MyIO (Node-RED + buffer SQLite) e faz a ponte entre o controlador VRF e a nuvem ThingsBoard. Não realiza acionamento elétrico direto sobre o VRF: toda a interação ocorre por rede (Modbus TCP/Ethernet) com o DCPF04, que mantém integralmente a responsabilidade física pelo barramento Daikin (DIII-Net via DCPA01).

🖥️ Composição de Hardware

  • Mini-PC industrial fanless (Intel NUC, ASUS PN, Advantech UNO) — ou Raspberry Pi 4/5 em case industrial — ou roteador edge industrial (Teltonika RUTX50 com 5G embutido)
  • 2–4 GB RAM, 64+ GB SSD
  • 2× Ethernet RJ45 (1 para Modbus DCPF04 + 1 para internet) — segmentos isolados por segurança
  • Modem 4G/5G embutido ou USB (com slot SIM) — failover de internet automático
  • WiFi opcional (apenas para configuração local — não usado em produção)
  • Alimentação: 12V DC ou 110/220V AC (depende do modelo)
  • Footprint: ~1U de rack ou prateleira de sala técnica · Fanless · Operação 24/7

📶 Topologia de Conectividade

O Gateway tem duas direções de comunicação independentes:

DireçãoDestinoTecnologia
LAN ↓ DCPF04 (Modbus TCP) RJ45 obrigatório
WAN ↑ ThingsBoard Cloud RJ45 + 4G/5G failover

Cenário recomendado: RJ45 cabeado como primário + 4G/5G como failover. Se cair a internet cabeada, o link móvel assume automaticamente — alinhado ao discurso de resiliência offline desta proposta.

⚡ Sem I/O Físico no VRF

O Gateway MyIO não tem nenhum I/O físico (digital, analógico, relé, 0–10V, 4–20mA) conectado ao VRF. Possui apenas conexão de rede Ethernet (RJ45).

[UIs Daikin VRF] │ DIII-Net (proprietário Daikin) ▼ [DCPA01] ← adaptador físico DIII-Net→Modbus │ RS485 (par trançado, A/B/GND) ▼ [DCPF04 Reiri] │ Ethernet RJ45 — Modbus TCP/502 ▼ [Gateway MyIO] ← só RJ45, nada físico no VRF │ Ethernet RJ45 — MQTT/HTTPS ▼ [ThingsBoard Cloud]

🔁 Entrada e Saída — apenas via Modbus

Apesar de não ter eletrônica de campo, o Gateway faz leitura e escrita — tudo encapsulado em Modbus TCP sobre a rede:

DireçãoOperação ModbusExemplos
Entrada (read) FC03 / FC04 Temperatura ambiente, setpoint, status on/off, código de erro, modo
Saída (write) FC06 / FC16 Ligar/desligar zona, mudar setpoint, trocar modo, fan speed

A "saída" do Gateway não é um pino que aciona algo — é um comando de rede que escreve num registro do DCPF04. Quem fisicamente comanda a UI Daikin é o próprio DCPF04 via DCPA01.

🛡️ Garantias do Fabricante Preservadas

A integração MyIO não invade nem altera a arquitetura elétrica/eletrônica original Daikin/Hitachi. Nenhum cabo é cortado, nenhum sensor é interceptado, nenhum sinal físico é injetado no barramento DIII-Net. O Gateway é apenas um cliente Modbus TCP do DCPF04 — uso documentado e suportado pela própria Hitachi no manual do equipamento (INS_DCPF04 Reiri for Office Touch).

Infraestrutura — Redundância, Segurança, Robustez
Seção 3 Infra

Três pilares de infraestrutura

Especificação completa para garantir operação 24/7 da integração VRF — com camadas de proteção e redundância proporcionais à criticidade do sistema de climatização para a operação do empreendimento.

🔄 Redundância

  • Energia: no-break (UPS) ≥ 30 min autonomia
  • Internet: RJ45 primário + 4G/5G failover automático
  • Buffer local: SQLite persiste todas as leituras em quedas
  • Cloud: ThingsBoard MyIO em cluster HA
  • Hardware spare: 1 Gateway extra cold standby (recomendado)
  • Time sync: múltiplos NTPs (pool + fallback local)

🔒 Segurança

  • Segmentação: Modbus em VLAN isolada da rede corporativa
  • Firewall outbound-only: zero portas expostas à internet
  • TLS/MQTTS (porta 8883 com certificado) para sync ThingsBoard
  • VPN WireGuard MyIO para suporte (sem SSH público)
  • OS hardenizado: Debian minimal + fail2ban + auto-update
  • APN privada se 4G/5G — sem IP público
  • Audit trail: log de todo comando write Modbus
  • Credenciais em vault (não hardcoded em flows)

🛡️ Robustez

  • Hardware industrial fanless: -10°C a 60°C, sem peças móveis
  • Watchdog hardware: reboot automático em travamento
  • systemd: Node-RED com auto-restart
  • Heartbeat: a cada 60s + alarme MyIO se Gateway sumir
  • Buffer retention policy: rotação automática (não enche disco)
  • Snapshot do sistema: restore <30 min em hardware spare
  • Runbook documentado: procedimentos de troca e troubleshooting

Para empreendimento comercial de uso contínuo, onde a climatização impacta diretamente o conforto de visitantes e lojistas, recomendamos o nível Recomendado. Ele entrega resiliência real (failover de internet, UPS, spare em prateleira, VLAN segregada) sem o custo do tier Enterprise — que só se justifica se o cliente já operar SLAs agressivos sobre o sistema de ar condicionado.

Plano de instalação — Gateway MyIO e infraestrutura

Serviço completo de implantação on-site do Gateway MyIO e da infraestrutura de suporte (rede, energia, cabeamento, segmentação). Executado em 5 etapas ao longo de aproximadamente 3–4 semanas, combinando trabalho remoto de preparação e 2 visitas presenciais à Melicidade.

ETAPA 1 Pré-instalação (remote) ~16h · 3–5 dias
  • Aquisição e recebimento do hardware (Gateway, UPS, switch, cabos, chip SIM)
  • Preparação da imagem do OS (Debian minimal + Node-RED + hardening)
  • Pré-configuração do Gateway: chaves MQTT, VPN WireGuard, certificados TLS
  • Teste de bancada MyIO: simulação Modbus TCP + validação sync ThingsBoard
  • Preparação do Gateway spare (snapshot idêntico)
ETAPA 2 Visita 1 — Levantamento on-site 1 dia · 8h
  • Mapeamento físico da sala técnica e do rack disponível
  • Confirmação do IP do DCPF04 e teste de porta 502 (Modbus TCP)
  • Levantamento de cobertura 4G/5G no local (teste com modem)
  • Definição de tomada elétrica, ponto de rede e layout final
  • Fotos, medições e plano detalhado de cabeamento
ETAPA 3 Visita 2 — Instalação física 1–2 dias · 16h
  • Montagem do Gateway no rack/prateleira (fixação DIN ou rack 1U)
  • Instalação e conexão do no-break (UPS) + teste de autonomia
  • Cabeamento de rede: LAN → DCPF04 e WAN → internet
  • Instalação do chip SIM + teste de failover 4G/5G
  • Configuração VLAN no switch + regras de firewall outbound
  • Etiquetação, documentação física e guarda do Gateway spare
ETAPA 4 Entrega Técnica & Go-Live 1 dia · 8h
  • Validação fim-a-fim: leitura Modbus → ThingsBoard → dashboard MyIO
  • Teste de escrita (on/off, setpoint, modo) em UI de teste isolada
  • Teste de queda de energia (UPS) e queda de internet (failover 4G)
  • Validação do buffer local: desconexão forçada + sync ao reconectar
  • Entrega de credenciais, runbook e treinamento do operador (1–2h)
ETAPA 5 Pós-instalação (30 dias) ~16h remote
  • Monitoramento ativo do Gateway (health check + alarmes proativos)
  • Ajustes finos de período de coleta e regras do Rule Engine
  • Suporte reativo a incidentes (canal dedicado MyIO)
  • Handover formal + aceite do serviço

Justificativa item a item — por que cada componente está na lista

Transparência total sobre a composição do CAPEX e OPEX. Cada linha indica a função do item, por que foi classificado como Essencial / Recomendado / Contingência, e qual é o risco caso o cliente opte por suprimi-lo ou adiá-lo.

💼 CAPEX — Hardware (one-shot)

Item Classificação Por que é necessário Risco de ausência / Alternativa
Gateway MyIO Essencial Cérebro da integração — executa polling Modbus, buffer local SQLite, sync ThingsBoard e cron de agendamentos. Fanless/industrial garante 24/7 sem manutenção. Sem ele não existe integração. Item insubstituível.
Gateway spare (cold standby) Recomendado Snapshot idêntico do primário guardado em prateleira. Em falha de hardware, restore fim-a-fim em <30 min. MTTR sobe de 30 min → 5–7 dias (RMA + aquisição + deslocamento). Inaceitável para uso 24/7.
Modem 4G/5G + chip Recomendado Failover automático se a internet cabeada cair. Habilita também cenário em que cliente não permite Gateway na rede corporativa. Queda de internet = dados ficam só no buffer; sem dashboard e sem comandos remotos. Pode ser eliminado se houver 2 ISPs redundantes.
No-break (UPS) Recomendado Protege Gateway contra queda de energia e picos. Autonomia ≥30 min preserva integridade do buffer SQLite durante shutdown controlado. Queda curta derruba o Gateway; picos podem danificar hardware; buffer pode corromper por shutdown abrupto.
Switch industrial L2 Recomendado* Permite VLAN dedicada para Modbus (isolado da rede corporativa) + SNMP para monitoramento de portas. Grau industrial para sala técnica. Pode virar Contingência se Gateway tiver 2× RJ45 suficientes (LAN+WAN) e topologia ponto-a-ponto ao DCPF04 — confirmar na Fase 0.
Cabos Cat6 blindado Essencial Blindagem (FTP/STP) reduz interferência eletromagnética típica de sala técnica (motores, inversores). Estabilidade do link Modbus TCP. Problemas intermitentes de conectividade (packet loss aleatório) difíceis de diagnosticar posteriormente.
Adaptador USB-RS485 Contingência Usado apenas se Fase 0 confirmar que o DCPF04 não expõe Modbus TCP — Gateway lê direto o barramento RS485 (Modbus RTU). Se cenário RTU se confirmar sem adaptador, integração inviável. Custo baixo (R$ 250) mantido como seguro preventivo.
Acessórios Essencial Fixação (DIN/rack), cabos de força, etiquetação padronizada, organizadores. Instalação profissional e rastreável. Instalação "amadora"; risco de desconexão acidental; troubleshooting futuro mais difícil.

Item Classificação Por que é necessário Risco de ausência / Alternativa
Plano dados M2M IoT 4G/5G Recomendado Mantém o chip ativo com plano dedicado. Planos M2M têm preços melhores que linha móvel convencional para telemetria 24/7 de baixo volume. Anexo D Glossário

Glossário — termos técnicos usados nesta proposta

Referência rápida dos termos, protocolos, equipamentos e conceitos de infraestrutura mencionados ao longo do documento. Organizado em dois grupos: (1) sistema VRF e comunicação com o DCPF04; (2) Gateway, rede, segurança e plataforma.

🏭 Sistema VRF & Comunicação

VRF Variable Refrigerant Flow — sistema de ar condicionado central onde uma unidade externa controla múltiplas unidades internas de forma independente por ambiente.
DCPF04 Controlador central Hitachi Reiri for Office Touch — "cérebro" do sistema VRF instalado na Melicidade. Gerencia todas as unidades internas Daikin.
DCPA01 Adaptador de protocolo Daikin → Modbus RTU. Traduz o barramento proprietário DIII-Net das UIs Daikin para Modbus.
DIII-Net Barramento proprietário Daikin para comunicação entre unidades externas e internas do VRF — não é publicamente documentado.
Modbus RTU/TCP Protocolo industrial aberto e amplamente documentado. Interface nativa do DCPF04 — RTU usa RS485; TCP usa Ethernet (porta 502).
FC03/FC04/FC06/FC16 Function Codes Modbus: FC03/FC04 leem registros (entrada); FC06 escreve 1 registro, FC16 escreve múltiplos (saída).
RS485 Padrão elétrico industrial para transmissão serial diferencial em par trançado (A/B/GND). Meio físico usado pelo Modbus RTU.
SMeter Medidor elétrico Daikin opcional por zona. Se instalado, permite coletar potência (W) e consumo (kWh) por ambiente.
iframe / túnel Modelo atual: a central MyIO age como proxy reverso, exibindo a UI web Hitachi dentro do dashboard. Dependente de conectividade contínua.

🌐 Gateway, Rede, Segurança & Plataforma

Gateway MyIO Mini-PC industrial fanless on-premise que faz a ponte entre o DCPF04 (Modbus TCP) e o ThingsBoard (nuvem MyIO). Sem I/O elétrico no VRF.
Node-RED Runtime de fluxos visuais em Node.js — orquestra o polling Modbus, o buffer local e o sync para a nuvem no Gateway MyIO.
Buffer local (SQLite) Banco embarcado no Gateway que armazena leituras durante quedas de conectividade — sincronizado quando a internet volta, sem perder timestamps.
MQTT / MQTTS Protocolo leve de mensageria IoT. MQTTS = MQTT sobre TLS (porta 8883). Usado para sync Gateway ↔ ThingsBoard.
ThingsBoard Plataforma IoT onde vivem os dashboards, relatórios, alarmes e Rule Engine (agendamentos) do MyIO.
Rule Engine Motor de regras declarativas do ThingsBoard — dispara ações (alarmes, RPC, notificações) em resposta a eventos de telemetria.
VLAN Segmentação lógica de rede num switch gerenciado — isola o segmento Modbus da rede corporativa por segurança.
WireGuard VPN Protocolo moderno de VPN (rápido e seguro) usado pela MyIO para suporte remoto ao Gateway — sem expor SSH à internet.
UPS / No-break Uninterruptible Power Supply — mantém o Gateway operando em quedas curtas de energia e protege contra picos/sub-tensão.
M2M IoT (4G/5G) Plano de dados móvel específico para telemetria 24/7 de baixo volume — preço melhor que linha móvel convencional, suporta APN privada.
Failover / Cold Standby Failover: troca automática para via secundária (ex: 4G assume se RJ45 cair). Cold standby: hardware spare desligado em prateleira, restaurado sob demanda.
SLA / CAPEX / OPEX SLA: Service Level Agreement (tempo de resposta contratual). CAPEX: investimento único (hardware). OPEX: custo recorrente (licenças, planos).
